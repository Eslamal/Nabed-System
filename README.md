# 🏥 Nabed Healthcare Ecosystem - Complete Digital Health Solution

![Platform](https://img.shields.io/badge/Platform-Android-green.svg) ![Language](https://img.shields.io/badge/Language-Kotlin-purple.svg) ![Architecture](https://img.shields.io/badge/Architecture-MVVM-blue.svg) ![Backend](https://img.shields.io/badge/Backend-Firebase-orange.svg)

**Nabed ** is a comprehensive, multi-application ecosystem designed to digitize the entire healthcare journey. Unlike standalone apps, Nabed connects Patients, Doctors, Receptionists, Pharmacies, and Delivery drivers in a real-time, synchronized network.

The system aims to streamline clinic operations, facilitate e-prescriptions, and ensure medication delivery within minutes.

---

## 🔄 System Architecture Overview

The ecosystem consists of **5 Native Android Applications** working in harmony:

1.  **Nabed Patient:** The user interface for booking and ordering.
2.  **Nabed Doctor:** The clinic management system (ERP & EMR).
3.  **Nabed Receptionist:** The front-desk operation manager.
4.  **Nabed Partner (Pharmacy):** The logistics and sales hub.
5.  **Nabed Delivery:** The last-mile delivery solution.

---

## 📱 1. Nabed Patient App
**"The Gateway to Healthcare"**

The patient app allows users to search for doctors, book appointments, receive digital prescriptions instantly, and order medicines from nearby pharmacies.

### Key Features
* **Smart Search:** Filter doctors by specialty, location, and rating.
* **Digital Prescription Wallet:** Receive and store prescriptions directly from the doctor.
* **Order Tracking:** Real-time tracking of medicine delivery drivers on the map.
* **Chat:** Communicate with clinics for inquiries.

## 👨‍⚕️ 2. Nabed Doctor App
**"The Clinical Brain"**

A mini-ERP system for doctors to manage their practice, finances, and patient records without paper.

### Key Features
* **In-Session Manager:** A dedicated flow for diagnosing, writing notes, and finishing visits.
* **Dynamic E-Prescription:** Build complex drug lists and send them to patients instantly.
* **Patient Linking:** Connect "Walk-in" patients to the cloud ecosystem via phone number.
* **Financial Dashboard:** Track income, expenses, and net profit.

## 👩‍💼 3. Nabed Receptionist App
**"The Operations Controller"**

A satellite application acting as the front desk for the clinic, managing flow without accessing sensitive medical data.

### Key Features
* **Gatekeeper System:** Review, approve, or reject pending booking requests.
* **Queue Management:** Manage walk-in patients and the live waiting list.
* **Linked Session:** Operates on the Doctor's data via secure "Collection Group Queries".
* **Unified Chat:** Responds to patients on behalf of the clinic.

## 🛠️ Technical Stack & Architecture

The entire ecosystem is built using **Native Android (Kotlin)** and follows **Modern Android Development (MAD)** practices.

### Core Technologies
| Category | Technology | Usage |
| :--- | :--- | :--- |
| **Language** | **Kotlin** | 100% of codebase. |
| **Architecture** | **MVVM** | Separation of concerns (Model-View-ViewModel). |
| **Concurrency** | **Coroutines & Flow** | Asynchronous operations & Reactive streams. |
| **Backend** | **Firebase Firestore** | Real-time NoSQL database. |
| **Auth** | **Firebase Auth** | Secure login/signup logic. |
| **Storage** | **Firebase Storage** | Storing profile pics & prescription images. |
| **Location** | **Google Maps SDK** | Map visualization & Tracking. |
| **UI** | **XML / ViewBinding** | Material Design 3 layouts. |
| **Local Data** | **DataStore** | Storing user preferences (Theme/Language). |

### 🚀 Key Technical Implementation Details

1.  **Atomic Transactions (Doctor App):**
    * Ending a visit triggers a `db.batch()` operation that updates the Doctor's record, the Patient's history, and clears the Queue simultaneously to ensure data integrity.

2.  **Cross-App Data Denormalization:**
    * Prescriptions are written directly to the `users/{id}/prescriptions` collection from the Doctor app, ensuring instant visibility for the Patient without complex queries.

3.  **Polymorphic Chat System:**
    * A custom-built Chat Adapter handling multiple ViewTypes (`SentText`, `ReceivedImage`, `SentAudio` with custom player) reused across Patient, Doctor, and Reception apps.

4.  **Satellite Architecture (Receptionist):**
    * The Receptionist app does not own data; it accesses the Doctor's sub-collections via dynamic linking, acting as a secure remote controller.

## 🔄 System Lifecycle (The "Happy Path")

1.  **Appointment:** Patient books via **Nabed Patient**. -> **Receptionist** approves it. -> Appears in **Doctor's** schedule.
2.  **Consultation:** Patient arrives. **Receptionist** checks them in. -> **Doctor** starts session, diagnoses, and writes E-Prescription.
3.  **Fulfillment:** Prescription appears in **Nabed Patient**. -> User orders medicine. -> **Nabed Pharmacy** receives order.
4.  **Delivery:** Pharmacy assigns driver. -> **Nabed Delivery** driver accepts. -> Patient tracks driver live until delivery.
